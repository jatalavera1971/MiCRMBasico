# Vibe CRM

Estructura base de la aplicación: Next.js (App Router, TypeScript) + Tailwind CSS v4. El paquete `convex` está instalado y listo para conectarse a Convex, pero todavía sin configurar (sin esquema, sin proveedor, sin funciones) — eso, y las pantallas del PRD, se construyen poco a poco a partir de las tareas de Linear (equipo `Joss`, proyectos `CRM-MVP` y `CRM-RESTOPRD`).

## Puesta en marcha

```bash
npm install
npm run dev
```

Abre [http://localhost:3000](http://localhost:3000).

## Desplegar

✅ **Actualizado 23 jul 2026 (JOS-60/61): login implementado, y las funciones de Convex de negocio ya exigen sesión válida — pero el riesgo original no queda cerrado del todo, léase con atención.** `/` es ahora la pantalla de login (email + contraseña); `src/app/(app)/layout.tsx` exige sesión para cualquier ruta bajo `(app)/`; y **todas** las funciones públicas de `clientes.ts`/`recordatorios.ts`/`interacciones.ts`/`dashboard.ts` (19 en total) exigen ahora un `token` de sesión válido (`requireSesion`, ver `convex/model/auth.ts`) — quien llame directamente al deployment de Convex sin un token ya no puede leer ni escribir nada de lo listado abajo.

**Lo que SÍ sigue siendo un riesgo real, sin resolver en este ticket:**
- **Sin scoping por rol**: cualquier usuario autenticado (Dueña o Comercial) puede llamar cualquiera de las 19 funciones — no hay distinción de permisos entre roles todavía. Confirmado como aceptable para esta ronda.
- **Recuperación de contraseña no funciona en absoluto** — no es una versión reducida, es un no-op total. `solicitarResetPassword` no envía ningún email, no genera ni guarda ningún token; el bottom sheet de "¿Olvidaste tu contraseña?" muestra "Correo enviado" pero no ha pasado nada server-side. No hay ningún camino real para recuperar una contraseña olvidada — la única vía hoy es re-sembrar el seed o editar `usuarios` a mano desde el dashboard de Convex.
- **Únicas cuentas existentes, credenciales de desarrollo compartidas**: sin P9 (JOS-62) no hay forma de dar de alta usuarios reales — las dos únicas cuentas son las que siembra `npx convex run seed:seedDemoData`: `marta@vibecrm.dev` (rol `duena`) y `diego@vibecrm.dev` (rol `comercial`), ambas con la contraseña `DevVibe2026!`. No usar en producción con datos reales sin cambiar esto primero.
- **Rate-limiting de login solo por email exacto, no por IP ni global**: `login` rechaza tras 10 intentos fallidos en 15 minutos, pero solo para ese email concreto — un atacante que rote emails inventados sigue forzando el hash PBKDF2 "señuelo" de forma indefinida. Riesgo de DoS de CPU de bajo coste si el deployment queda público, no resuelto aquí.
- **`usuarios.by_email` no tiene unicidad impuesta por Convex** (no hay constraint nativo) — el seed siempre parte de una tabla vacía, así que no es explotable hoy, pero cualquier alta futura (JOS-62/P9) deberá validar duplicados explícitamente antes de insertar.
- **Sesiones fijas de 30 días** desde el login, sin renovación deslizante ni invalidación remota ("cerrar sesión en todos los dispositivos" requeriría P9/P10). Se limpian de la tabla `sesiones` vía un cron diario (`convex/crons.ts`, acotado a 500 filas/ejecución).
- **Botón "Cerrar sesión" temporal** en `Sidebar`/`TabBar` (footer del menú) — sin P10 (JOS-63) no existe todavía una pantalla de perfil propiamente dicha; ese botón se sustituirá cuando se construya.

**Lo que queda del riesgo histórico original** (aplicable ahora solo a usuarios ya autenticados, Dueña o Comercial, sin distinción de rol — antes de JOS-60/61 era literalmente cualquiera con la URL):
- **Leer**: `listarClientes` (JOS-10/13/45, 13 jul 2026) devuelve `nombre`, `email`, `telefono`, `empresa`, `canal_preferido`, `fase`, `prioridad` y `fecha_ultimo_contacto` de hasta 500 clientes; `listarClientesInactivos` (JOS-25, ampliada JOS-26 el 19 jul 2026) devuelve `nombre`/`empresa`/`fecha_ultimo_contacto`/`prioridad`/`diasSinContacto` de **todos** los clientes inactivos existentes, sin cap (`prioridad` ya era pública vía `listarClientes`/`obtenerPipeline`, `diasSinContacto` es derivado de `fecha_ultimo_contacto`, ya público — no amplía el riesgo); `obtenerCliente` (JOS-11, 15 jul 2026) devuelve esos mismos campos de un cliente concreto por id — no es un salto de riesgo nuevo, ya que los ids son públicos desde `listarClientes`, se documenta para que la lista quede completa. Ninguna expone el documento completo, pero sí más campos (incluido `telefono` y la actividad comercial reciente) que antes de JOS-10.
- **Escribir**: `crearCliente` (JOS-12) permite a cualquier usuario autenticado crear clientes nuevos sin límite ni verificación adicional — mitigado solo con límites de longitud server-side (nombre/email/empresa/teléfono), sin rate-limit ni captcha.
- **Actualizar recordatorios**: `marcarComoHecho` permite a cualquier usuario autenticado marcar cualquier recordatorio como hecho conociendo su id, sin scoping por quién lo creó.
- **Recordatorios manuales (JOS-22, 17 jul 2026)**: `crearRecordatorio` permite a cualquier usuario autenticado (sin scoping por rol) crear recordatorios (fecha + motivo) para cualquier cliente conociendo su id, y `actualizarRecordatorio` permite editar fecha/motivo de uno ya existente — primera vía de escritura PÚBLICA DIRECTA sobre esta tabla (antes solo existía el efecto automático de `crearInteraccion`). Riesgo ampliado aceptado explícitamente el 17 jul 2026, con mitigaciones server-side: `motivo` máximo 200 caracteres, `fecha` validada como `YYYY-MM-DD` real (sin restricción de pasado/futuro — decisión de producto deliberada), y `actualizarRecordatorio` verifica que el recordatorio pertenezca al `clienteId` indicado y siga `pendiente` antes de aceptar el cambio — esto último es una mitigación de CONTRATO, no un control de autorización real por usuario: cualquier usuario autenticado que conozca ambos ids podría seguir editando, sin importar quién creó el recordatorio.
- **Actualizar clientes (JOS-11, 15 jul 2026)**: `actualizarCliente`, `actualizarCanalPreferido` y `actualizarPrioridad` permiten a cualquier usuario autenticado (sin scoping por rol) modificar la PII (nombre/email/empresa/teléfono), el canal preferido o la prioridad de cualquier cliente conociendo su id, sin scoping por rol ni por quién lo creó. Salto de riesgo mayor que crear/leer: antes solo se podía añadir datos o marcar un recordatorio hecho, ahora se puede sobrescribir cualquier cliente ya existente.
- **Cambiar fase de pipeline (JOS-14/15, 17 jul 2026)**: `actualizarFase` permite a cualquier usuario autenticado (sin scoping por rol) mover cualquier cliente a cualquiera de las 6 fases del pipeline (`lead`/`cualificacion`/`primera_llamada`/`propuesta_enviada`/`negociacion`/`cerrado`) conociendo su id, sin scoping por rol. Mismo patrón/riesgo ya aceptado que `actualizarPrioridad`: mitigado solo porque el valor está restringido por el union type del schema, sin validación de negocio adicional (no hay restricción de transición, se puede mover a cualquier fase, adelante o atrás). La nueva query `obtenerPipeline` (vista Pipeline, P6) no amplía superficie de lectura: agrupa por fase los mismos campos que `listarClientes` ya expone públicamente, pero con una proyección propia más reducida (sin `email`/`telefono`/`canal_preferido`, que P6 no consume).
- **Eliminar clientes (JOS-11, 15 jul 2026)**: `eliminarCliente` borra permanentemente cualquier cliente y, en cascada, todos sus `recordatorios` e `interacciones` — cualquier usuario autenticado puede hacerlo, sin scoping por rol ni confirmación más allá de la del propio navegador. Es la mutation pública más destructiva del proyecto hasta ahora — riesgo ampliado aceptado explícitamente el 15 jul 2026 (ronda de revisión previa a la implementación), mismo patrón que el resto de esta sección.
- **Reactivar cliente inactivo (JOS-26, 19 jul 2026)**: `reactivar` permite a cualquier usuario autenticado (sin scoping por rol) actualizar `fecha_ultimo_contacto` a la hora actual de cualquier cliente conociendo su id, sin scoping por rol y **sin crear un registro de interacción** — deliberadamente distinta de `crearInteraccion`. Mitigación de CONTRATO (no de autorización real, mismo matiz que `actualizarRecordatorio`): la mutation valida server-side que el cliente siga cumpliendo el predicado de inactividad (`fecha_ultimo_contacto` presente y a más de 7 días — un cliente nunca contactado no cuenta como inactivo, mismo criterio que JOS-25) antes de aplicar el cambio — rechaza la operación si el cliente ya no es inactivo. Sin esa validación, cualquier `clienteId` válido serviría para tocar la fecha de un cliente activo, cerrado o recién creado.
- **Interacciones (JOS-18/19/20/21, 16 jul 2026)**: `crearInteraccion` permite a cualquier usuario autenticado (sin scoping por rol) escribir notas de texto libre sobre las conversaciones con cualquier cliente (conociendo su id), y `listarInteracciones` permite leerlas — superficie nueva y más sensible que la PII de contacto ya expuesta, ya que el contenido es texto libre sin estructura. Riesgo ampliado aceptado explícitamente el 16 jul 2026 (ronda de revisión previa a la implementación), con estas mitigaciones técnicas server-side (nunca solo en el formulario del cliente): `notas` máximo 2000 caracteres, `proximo_paso_texto` máximo 200, la fecha de la interacción se rechaza si es futura (margen 24h) o anterior al alta del cliente, `proximo_paso_fecha` se valida como fecha de calendario real, y `listarInteracciones` está acotada a las 500 interacciones más recientes por cliente (mismo cap que `listarClientes`, evita que la escritura pública sin límite pueda inflar el historial de un cliente concreto sin control).
- **Límite funcional, no de seguridad**: la búsqueda de `/clientes` (JOS-13) filtra en el navegador sobre los hasta 500 clientes que devuelve `listarClientes` — si la tabla creciera más allá de eso (posible precisamente por la escritura pública sin límite), la búsqueda dejaría de cubrir el resto sin que se note. Revisar al abordar JOS-31 (QA de rendimiento 500+).
- **Límite funcional, no de seguridad (recordatorios, 16-17 jul 2026)**: `crearInteraccion` (vía "próximo paso") y, desde JOS-22, `crearRecordatorio` directamente, pueden crear recordatorios públicos — antes de JOS-18/21 esta tabla solo crecía por seeds internos, nunca por una mutation pública. `listarSeguimientosHoy` (P1) y `listarRecordatoriosPendientes` (ficha) están acotadas a 500 filas (`.take(500)`); esta última usa el índice compuesto `by_cliente_estado_fecha` (cliente_id, estado, fecha) para que el cap conserve de verdad los pendientes MÁS PRÓXIMOS de ese cliente, no simplemente los más recientemente creados. Si el total de recordatorios pendientes (globales para P1, o de un cliente para la ficha) superara 500, dejaría de ser exhaustivo sin que se note — límite funcional aceptado, no hay paginación real.
- **Límite funcional, no de seguridad (pipeline, 17 jul 2026)**: `obtenerPipeline` (P6) usa el mismo cap `.take(500)` que `listarClientes`, ordenado por `fecha_alta` — si la tabla de clientes supera esos 500, el pipeline deja de mostrar el resto sin que se note (mismo límite ya documentado, no se resuelve paginación real en esta tarea). La pantalla lo comunica explícitamente ("Mostrando hasta 500 clientes") en vez de presentar el cap como si fuera el total real del pipeline.
- **Límite funcional, no de seguridad (clientes inactivos, 19 jul 2026)**: a diferencia del resto de listados de esta sección, `listarClientesInactivos` **no** capa la query (deliberado: un `.take()` antes de excluir los clientes nunca contactados podría devolver menos inactivos reales de los que hay, y esta misma función alimenta el contador exacto del banner de P1). El cap de 500 filas de `/inactivos` se aplica solo en el frontend (`InactivosListClient`), a efectos de cuántas filas se pintan en el DOM — la query en sí sigue leyendo todos los clientes inactivos existentes en cada visita, sin límite. Con una tabla muy grande esto es un coste de lectura mayor que el resto de listados (que sí acotan en el propio query), aceptado explícitamente por ser la opción que preserva la corrección del conteo.

Se decidió conscientemente activar el despliegue de todos modos (2026-07-12, riesgo de escritura pública reafirmado el 13-07-2026 al ampliar la lectura, el 15-07-2026 al añadir actualización y borrado permanente de clientes, el 16-07-2026 al añadir notas de texto libre sobre clientes, el 17-07-2026 al añadir creación/edición directa de recordatorios y edición directa de fase de pipeline, el 19-07-2026 al añadir reactivación directa de clientes inactivos, y el 23-07-2026 al cerrar el acceso público con JOS-60/61 — login + guard de sesión en las 19 funciones de negocio). El riesgo de acceso público directo a los datos de clientes queda cerrado; el riesgo de falta de scoping por rol entre usuarios autenticados, y el resto de puntos listados arriba, quedan explícitamente abiertos para un ticket de hardening futuro. Railway está conectado a este repo por GitHub para auto-deploy — cualquier push a `main` dispara un despliegue real.

- **Railway**: `railway.json` ejecuta `npx convex deploy --cmd 'npm run build'` (despliega las funciones de Convex a producción, fijando `NEXT_PUBLIC_CONVEX_URL` para el build) y arranca con `npm run start`. Railway asigna `$PORT` automáticamente.
  - `convex/_generated` **sí se versiona en git** (siguiendo la recomendación oficial del propio `npx convex codegen --help`: "should be committed to the repo") — `--cmd` se ejecuta *antes* de que `convex deploy` regenere esa carpeta, así que si no estuviera commiteada, `npm run build` fallaría en cualquier clone limpio (incluido Railway) con `Module not found` sobre `convex/_generated/api`. Si cambias el schema o las funciones, corre `npx convex codegen` (o `npx convex dev`) y commitea los cambios en `convex/_generated`.
  - **Requisito ya configurado:** deployment de producción de Convex creado (`diligent-chipmunk-930`, región `eu`) y `CONVEX_DEPLOY_KEY` puesto en el proyecto de Railway (nunca en un fichero versionado — se generó con `npx convex deployment token create ... --prod` y se pasó directo a `railway variable set --stdin`, sin exponerlo en ningún log).
- **Convex**: `npx convex deploy` para producción (mismo comando que usa Railway) — la primera vez, crea antes el deployment de producción en la región correcta con `npx convex deployment create --type prod --region eu --default`.
